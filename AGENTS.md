# AGENTS.md — 奶茶实验室 Codex 工作守则

## 1. 项目身份

本项目暂定名为：

**奶茶实验室**

当前仓库名：

**milktealab**

这是一款原创饮品研发经营模拟游戏。

核心定位不是“奶茶店打工模拟”，也不是复刻任何已有游戏。

玩家身份是：

- 奶茶店老板
- 饮品研发师
- 配方实验狂人

项目核心乐趣是：

> 自由调配饮品 → 试喝员毒舌反馈 → 保存/优化配方 → 后续上架营业验证市场反应

一句话方向：

> 不是重复手动摇杯，而是发明一杯奇怪奶茶，看它能不能卖爆

---

## 5. 项目设计原则

### 5.1 不是打工模拟

不要把游戏做成手忙脚乱接单、摇杯、出杯的打工模拟。

核心不是操作速度。

核心是：

- 研发
- 调配
- 试喝
- 命名
- 保存
- 上架
- 看市场反馈

---

### 5.2 配方系统必须自由

不要做成固定槽位。

错误方向：

- 茶底只能选一个
- 奶类只能选一个
- 水果只能选一个
- 小料只能选一个

正确方向：

**成分池 + 比例系统**

分类只用于 UI 分组和筛选，不限制选择数量。

玩家应该可以自由做出：

- 红茶 + 牛奶 + 珍珠
- 绿茶 + 柠檬 + 蜂蜜 + 气泡水
- 乌龙茶 + 厚乳 + 奶盖 + 海盐
- 气泡水 + 淡奶油 + 榴莲 + 咖啡
- 多种茶混合
- 单一原料极端高比例事故饮品

---

### 5.3 茶底和乳基必须拆开

“奶茶”不是一个基础原料。

奶茶是：

> 茶 + 乳基 + 其他配料

不要把“奶茶”和红茶、绿茶、乌龙茶并列为茶底。

---

### 5.4 试喝反馈是项目灵魂

试喝反馈不能像普通评分器。

不能像 AI 客服。

不能只写“味道不错”“口感一般”。

反馈要：

- 毒舌但可爱
- 有画面感
- 能体现系统真的读懂了配方
- 正常组合要能夸得具体
- 离谱组合要吐槽得有记忆点
- 事故组合要好笑但不影响玩家继续实验

风格参考：

- 榴莲味很有主见，已经把其他材料全部开除了
- 气泡和奶油在杯子里打架，试喝员正在思考人生
- 吸管刚插进去就提交了辞职信
- 我本来想骂，结果喝完沉默了，居然有点合理

不要把反馈写成正式报告腔。

---

### 5.5 隐藏配方不是免死金牌

未来加入隐藏配方时，不能让隐藏配方直接覆盖味觉好坏。

隐藏配方判断应分三层：

1. 身份识别
   判断是否命中隐藏配方、是否解锁图鉴、属于什么系列
2. 味觉审判
   继续判断比例是否翻车、是否奶脂过载、是否吸管阻力事故、是否组合冲突
3. 合成结果
   如果命中隐藏配方且比例正常，显示隐藏饮品名并给正常/较好评价
   如果命中隐藏配方但比例离谱，仍然低分并吐槽翻车版本
   灾难隐藏款可以低分，但仍然解锁图鉴

核心原则：

> 发现配方有奖励，优化比例有深度，翻车版本也有笑点

---

### 5.6 内容不能全是整活

整活饮品负责记忆点和传播。

但项目不能只有灾难款。

长期内容生态需要包括：

- 经典稳定款
- 清爽水果茶款
- 花果茶款
- 浪漫 / 拍照款
- 健康自律款
- 打工续命款
- 猎奇灾难款
- 隐藏收藏款

正常好喝的饮品是经营系统的基础。

灾难饮品是传播和截图价值。

两者都需要。

---

### 5.7 默认展示味觉体验，不默认展示后台数值

测试期可以展示详细属性条，方便调试。

正式版默认不应让玩家只盯着数值玩成“配表格”。

正式版更适合优先展示：

- 饮品类型
- 综合评分
- 试喝员评价
- 适合客群
- 风味标签
- 模糊等级

详细数值可以作为：

- 风味分析仪
- 高级模式
- 折叠详情
- 实验室设备解锁

原则：

> 默认展示味觉体验，不默认展示后台数值

## 6. 版本规则

项目使用四段版本号：

```text
A.B.C.D
```

含义：

- A：正式/测试阶段
  - 0 = 测试期
  - 1 = 正式发布后
- B：大功能阶段 / 大模块
  - 0 = 配方实验室
  - 1 = 经营循环
  - 2 = 员工/养成等，后续再定义
- C：重大优化或底层逻辑变更
- D：同一重大优化下的小迭代 / 修补

示例：

```text
v0.0.3.2
```

表示：

- 测试期
- 配方实验室阶段
- 第 3 个重大优化方向
- 第 2 个小修版本

每次修改版本时，需要同步更新：

- UI 顶部版本号
- `docs/VERSION_LOG.md`

## 8. 工程原则

### 8.1 静态网页优先

当前项目应保持简单静态网页结构。

不要主动引入复杂工程栈。

当前适合：

- HTML
- CSS
- 原生 JavaScript
- ES Modules
- GitHub Pages 静态部署

除非任务明确要求，否则不要引入：

- React
- Vue
- Svelte
- TypeScript
- Vite
- Webpack
- Electron
- Tauri
- 后端服务
- 数据库
- 登录系统

---

### 8.2 不要炫技式重构

不要为了“架构漂亮”而大改逻辑。

不要一次性重写整个项目。

不要把简单逻辑拆得过碎。

优先级：

1. 保持功能稳定
2. 保持现有体验
3. 降低未来维护风险
4. 再考虑代码美观

---

### 8.3 game.js 不应继续无限膨胀

长期目标是让 `game.js` 变成轻入口文件。

理想职责：

- 导入数据
- 初始化状态
- 初始化 UI
- 绑定事件
- 启动首次渲染

不应长期承担：

- 所有味觉判断
- 所有评分计算
- 所有反馈选择
- 所有 DOM 渲染
- 所有事件逻辑
- 所有存档逻辑

---

### 8.4 v0.0.5.x 味觉系统治理优先级

v0.0.5.x 前几个版本应优先继续治理味觉系统代码结构和规则数据化，不急着调数值 / 调味觉表现。

优先治理：

1. `drinkTypeAnalyzer.js` 的饮品类型识别数据化。
2. `feedbackEngine.js` 的反馈标签化 / 文案池规则化。
3. `proportionAnalyzer.js` 的比例段规则表。
4. `accidentAnalyzer.js` 的事故规则表。
5. 金标样本 / 回归样本地基。

在这些结构稳定前，不要开始大规模调味觉数值。

### 8.5 if 治理原则

项目不是要彻底消灭 if，而是要把 if 从“到处乱接线”升级成“总电闸 + 规则表驱动”。

少量中枢 if 可以负责调度、判断规则是否触发、处理优先级和安全阈值；具体味觉内容应尽量放入规则表、属性表、文案池和测试样本。

后续注意：

- `tasteJudge.js` 已经变薄，不要让它重新变胖。
- 不要让 `drinkTypeAnalyzer.js`、`proportionAnalyzer.js`、`accidentAnalyzer.js` 或 `feedbackEngine.js` 变成新的巨型垃圾桶。
- 新增普通组合优先进 `combinationRules.js`。
- 新增原料属性优先进 `ingredientTasteProfiles.js`。
- 新增反馈优先进文案池 / 标签系统。
- 新增类型识别规则未来应进入类型规则表。
- 新增比例规则未来应进入比例段规则表。

### 8.6 省 token / 大项目 Codex 工作流

随着项目变大，每轮 Codex 任务必须明确“允许读取 / 修改的文件范围”，避免全项目扫描和无关重构。

长期原则：

1. 不要让 Codex 每次重新考古整个项目。
2. 每轮任务前应明确本轮目标、允许读取的文档、允许修改的文件、不允许碰的文件。
3. 稳定模块应尽量冻结，除非本轮明确需要，不要反复读取和修改。
4. 大任务先让 Codex 输出计划，包括准备读哪些文件、准备改哪些文件、哪些文件不碰、风险点是什么，再执行。
5. 任务越往后越要小，不要用“继续优化奶茶实验室”这种大范围指令。
6. 以后优先让 Codex 阅读本轮相关的 docs 和 2–5 个核心文件，而不是全仓库扫描。
7. 自动测试和金标样本是省 token 的工具，可以减少反复人工判断和大段解释。

错误方式：

```text
Codex，继续优化奶茶实验室。
```

正确方式：

```text
Codex，本轮只读 docs/TASTE_SYSTEM_DESIGN.md、drinkTypeAnalyzer.js 和未来类型规则表。本轮只把饮品类型识别从 if 迁移到规则表，不改评分、不改反馈、不改 UI。完成后跑金标样本。
```

### 8.7 云记忆与本地 docs 分工

云端长期记忆不是项目仓库，只保留少量跨对话必须自动调用的长期原则、项目定位、核心资产方向、关键工作流、重要边界和用户长期偏好。

版本流水账、commit hash、tag、单次验收结果、具体改了哪些文件、某次测试是否通过、备份路径等，应写入本地 docs、`docs/VERSION_LOG.md` 或 GitHub/tag，不要写进云端长期记忆。

本地 docs 才是项目正本。

一句话：

> 云记忆是导航仪，不是仓库。

## 10. 测试要求

每次修改后，至少检查：

### 基础操作

- 页面能正常打开
- 原料按钮可点击
- 点击未加入原料会加入当前杯子
- 再点已加入原料会移除
- 已选按钮高亮正常
- ✓ 显示正常
- 清空杯子正常
- 随机乱摇正常

### 比例逻辑

测试：

```text
红茶 50%
牛奶 30%
珍珠 20%
```

把红茶拖到 40%。

预期：

```text
红茶 40%
牛奶 30%
珍珠 20%
总量 90%
```

确认：

- 其他原料不自动变化
- 总量可以低于 100%
- 总量不能超过 100%

### 试喝报告

至少测试：

- 红茶 + 牛奶 + 珍珠
- 绿茶 + 柠檬 + 蜂蜜 + 气泡水
- 茉莉茶 + 荔枝 + 桃子 + 蜂蜜 + 纯水
- 榴莲 + 牛奶
- 淡奶油极高比例组合
- 芋泥 + 气泡水
- 多种茶类混合
- 随机乱摇 5 次

确认：

- 无 `undefined`
- 无 `[object Object]`
- 无空白报告
- 无控制台 error

### 保存配方

确认：

- 保存正常
- 刷新后仍在
- 载入正常
- 载入后左侧高亮同步
- 删除保存配方正常

## 12. 最重要提醒

不要把这个项目当普通 CRUD 小工具。

这是一个长期原创游戏项目。

每次修改前都要先判断：

1. 是否符合“饮品研发模拟”的核心定位？
2. 是否破坏了自由配方系统？
3. 是否削弱了试喝反馈的灵魂？
4. 是否让玩家体验变成数值表格？
5. 是否引入了不必要的复杂度？
6. 是否会让未来维护更困难？

如果任务要求和本文件冲突，请优先提醒用户确认，不要直接强行修改。
