# 奶茶实验室 Project Rules

## 1. 项目定位

《奶茶实验室》是原创饮品研发经营模拟游戏。灵感来自《疯狂摇摇杯》，但不能复刻其原名、素材、UI、配方表或原文案。

最终目标是手游 + Steam 端游。当前 Web 技术只是快速原型手段。

《奶茶实验室》不是手忙脚乱接单摇奶茶的游戏，而是饮品研发、配方实验、试喝吐槽、经营验证的游戏。

文档层级、冲突裁决和更新归属以 `docs/DOCS_SOURCE_OF_TRUTH.md` 为准。

## 2. 当前阶段

当前项目处于 P0 docs recovery / pause mode，直到 ChatGPT + 用户确认解除。

P0 未解除前：

- 不开 v0.0.8.x。
- 不做 batch content。
- 不生成新 ID / feedbackTag / triggerMetric / ingredient profile。
- 不做 generated severity。
- 不做 shadow / partial / active takeover。
- 不让 Codex 发明机制概念。
- 不把 v0.0.7.x 视为安全 closure。

历史阶段：

- v0.0.5.x：现有核心系统 ID 化 / 去显示文案主键 / 平台无关数据地基。
- v0.0.6.x：三层属性 / profile / summary 地基。
- v0.0.7.x：原计划进入 severity / threshold / scoreMultiplier / 内容管线调参，但当前因判定模型正本与 AI_CONTEXT 污染进入 P0 recovery。

## 3. P0 pause mode 禁止事项

P0 期间不做任何玩法 / 内容 / runtime 扩张；只处理 ChatGPT + 用户明确批准的 P0 recovery 范围。

- 不新增玩法、原料、隐藏配方、营业系统、顾客系统、员工系统。
- 不改 UI 视觉或交互，除非任务明确要求。
- 不为了单个案例写死配方特判。
- 不把温度、冰量、糖度做成完整可玩功能，除非任务明确要求。
- 不新增 batch content、stable ID、feedbackTag、triggerMetric 或 ingredient profile。
- 不接入 generated severity、shadow、partial takeover 或 active takeover。
- 不让 Codex 自行发明机制概念。
- 不把 v0.0.7.x 写成已经安全 closure。

## 4. 配方系统长期规则

配方系统是成分池 + 比例系统，不是每类只能选一个。

茶底和乳基分开，“奶茶”不是茶底，而是茶 + 乳基的结果。

当前判定模型以 `docs/TASTE_DECISION_MODEL.md` 为准；本文只做入口提醒，不复制完整长设计。

旧线性优先级：

```text
极端比例事故 > 质地事故（奶脂过载/吸管阻力） > 冲突组合 > 正常好组合 > 普通分类
```

已废弃为 deprecated / historical rough rule only，不能再作为当前判定模型正本。

当前正确判定层级是：

```text
特殊服务事故 > 质地事故 > 味觉事故 > 风味冲突 > 正常好组合 > 普通分类
```

判定优先级不等于扣分严重度；`priorityBand` 不等于 `severityLevel`；`severityHint` 不等于 `scoreMultiplier`。

反 if 地狱是全判定模型原则，不是 flavor 层专属原则。特殊服务事故、质地事故、味觉事故、风味冲突、正常好组合、普通分类，都不能长期写成具体原料组合 if 地狱。

隐藏配方以后走身份识别线，不是免死金牌；事故仍然应该优先被识别，但最终扣分和反馈强度应由 severity / evidence / triggerMetric / thresholds / weights 等后续层决定。

测试期可以显示完整属性条；正式版默认不展示所有后台数值。

宽屏桌面优先，竖屏只是兼容。

## 5. Vibe Coding 工作流

用户不是程序员。Codex 的任务单必须清晰、可执行、低歧义。每次修改后必须输出变更摘要和自检结果。

任务单应尽量包含：

- 版本号
- 阶段名
- 目标
- 本轮不做事项
- 范围边界
- 执行步骤
- 自检清单
- 回滚 / 备份要求

随着项目变大，每轮 Codex 任务必须明确“允许读取 / 修改的文件范围”，避免全项目扫描和无关重构。

长期原则：

1. 不要让 Codex 每次重新考古整个项目。
2. 每轮任务前应明确本轮目标、允许读取的文档、允许修改的文件、不允许碰的文件。
3. 稳定模块应尽量冻结，除非本轮明确需要，不要反复读取和修改。
4. 大任务先让 Codex 输出计划，包括准备读哪些文件、准备改哪些文件、哪些文件不碰、风险点是什么，再执行。
5. 任务越往后越要小，不要使用“继续优化奶茶实验室”这种大范围指令。
6. 以后优先让 Codex 阅读本轮相关的 docs 和 2–5 个核心文件，而不是全仓库扫描。
7. 自动测试和金标样本是省 token 的工具，可以减少反复人工判断和大段解释。

错误方式：

```text
Codex，继续优化奶茶实验室。
```

正确方式：

```text
Codex，本轮只读 docs/TASTE_SYSTEM_DESIGN.md、drinkTypeAnalyzer.js 和未来类型规则表。本轮只把饮品类型识别从 if 迁移到规则表，不改评分、不改反馈、不改 UI。完成后跑金标样本。
```

## 6. 反 if 地狱规则

味觉系统未来必须优先数据驱动，避免大量具体配方硬编码 if/else。

项目不是要彻底消灭 if，而是要把 if 从“到处乱接线”升级成“总电闸 + 规则表驱动”。

少量中枢 if 可以负责调度、判断规则是否触发、处理优先级和安全阈值；具体味觉内容应尽量放入规则表、属性表、文案池和测试样本。

新增味觉规则时，优先放入：

- 原料属性库
- 协同 / 冲突规则
- 饮品模式规则
- 金标样本
- 反馈文案库
- 客群偏好数据

如果必须新增硬编码特判，必须在报告里说明原因和数量。

后续重点防线：

- `tasteJudge.js` 已经变薄，不要让它重新变胖。
- 不要让 `drinkTypeAnalyzer.js`、`proportionAnalyzer.js`、`accidentAnalyzer.js` 或 `feedbackEngine.js` 变成新的巨型垃圾桶。
- 新增普通组合优先进 `combinationRules.js`。
- 新增原料属性优先进 `ingredientTasteProfiles.js`。
- 新增反馈优先进文案池 / 标签系统。
- 新增类型识别规则未来应进入类型规则表。
- 新增比例规则未来应进入比例段规则表。

## 7. 模块边界长期规则

《奶茶实验室》未来应按系统模块拆开推进，不要让 Codex 一次性“生成整个游戏”。

长期系统边界：

- 味觉系统：负责判断这杯饮品是什么、好不好、有什么事故、有什么属性、有什么标签、适合哪些客群。
- 反馈系统：负责根据味觉系统输出的标签、事故、类型和属性，选择合适的试喝反馈文案。
- 配方实验室系统：负责当前杯子、原料选择、比例调整、随机乱摇、保存配方等。
- 隐藏配方 / 图鉴系统：负责特殊组合识别、图鉴解锁、收藏进度和隐藏饮品身份。
- 顾客系统：负责根据饮品结果和顾客偏好，判断不同客群是否购买、如何评价。
- 经营系统：负责上架饮品、成本、定价、销量、利润、口碑、日结等。
- 员工 / 设备 / 升级系统：负责员工效率、疲劳、设备加成、升级效果等。
- UI 展示层：负责展示状态和交互，不重新判断味觉。
- 存档系统：负责保存配方、图鉴、经营进度和玩家状态。

关键边界：

- 味觉系统判断“柠檬牛奶是不是事故”。
- 反馈系统只根据味觉标签选择怎么吐槽。
- 顾客系统读取味觉结果和客群偏好，不重新判断味觉事故。
- 经营系统读取顾客购买和饮品数据，不重新计算味觉。
- UI 只展示结果，不承担业务判断。

模块化不是简单分文件，而是每个系统有清楚职责、输入和输出。

以后 Codex 任务应尽量一次只动一个系统。模块化可以减少 Codex 每次全项目扫描；稳定模块不要反复打开和改动，系统接口文档会越来越重要。

## 8. 云记忆与本地 docs 分工

云端长期记忆不是项目仓库，只保留少量跨对话必须自动调用的长期原则、项目定位、核心资产方向、关键工作流、重要边界和用户长期偏好。

版本流水账、commit hash、tag、单次验收结果、具体改了哪些文件、某次测试是否通过、备份路径等，应写入本地 docs、`docs/VERSION_LOG.md` 或 GitHub/tag，不要写进云端长期记忆。

本地 docs 才是项目正本。

一句话：

> 云记忆是导航仪，不是仓库。

## 9. 稳定版本规则

每个稳定版本需要：

- 更新 VERSION_LOG
- 更新 AI_CONTEXT
- 确认 git status
- 提交 commit
- 打 tag
- 必要时做本地备份
- 输出可读的测试报告

### Candidate 冻结前版本号同步规则

从 v0.0.5.7 起，只要准备创建 `vX.X.X.X-candidate` tag，页面顶部显示版本号必须已经同步到该 candidate 版本。

即使该版本只是文档、测试脚本、回归样本、工程地基或内部规则更新，只要它要作为 candidate 冻结，也必须保证玩家 / 测试者在页面上看到的版本号与 tag 版本一致。

冻结前浏览器验收必须检查页面顶部版本号。若页面显示版本号落后于准备打 tag 的版本，必须先修正版本号、提交并 push，再重新执行冻结前复查，最后再打 candidate tag。

禁止在页面版本号与 candidate tag 不一致的情况下打 tag。

说明：v0.0.5.6-candidate 已存在一个页面显示仍为 v0.0.5.5 的小瑕疵；该版本不重打 tag，后续版本按本规则执行。

### AI_CONTEXT 持续更新规则

`docs/AI_CONTEXT.md` 是新 ChatGPT / Codex 对话恢复项目上下文的核心入口，不是完整版本记录，但必须保持可接续。

每次出现以下情况，都必须检查 `docs/AI_CONTEXT.md` 是否需要同步：

- candidate 版本冻结后
- 当前版本状态发生变化
- 下一步计划发生变化
- 新增长期工程原则
- 新增会影响后续 Codex 任务的工作流规则
- 重要模块路线发生变化

如果需要更新，只写新对话接续所需摘要和长期规则，不写详细流水账。详细版本记录仍写入 `docs/VERSION_LOG.md`，详细设计写入对应专题 docs。

禁止只更新 `VERSION_LOG`、`TASTE_SYSTEM_DESIGN`、`TEST_CASES` 等文件，却让 `AI_CONTEXT` 长期滞后。

## 10. 自动化验收与失败自救规则

Codex 每次完成版本修改后，应优先尝试自动化验收，尽量减少用户手动测试负担。

如果第一次自动化验收失败，Codex 不应立刻把全部测试交给用户，而应先自行做一次低风险恢复尝试，例如：

- 确认本地静态服务是否仍在运行
- 重启静态服务
- 换一个可用端口
- 确认访问 URL 是否正确
- 重新打开页面
- 重连可用的浏览器自动化工具

如果低风险恢复后仍失败，Codex 才应报告：

- 失败原因
- 已尝试的恢复步骤
- 已完成的替代检查
- 用户仍需手动检查的最小 P0 清单

## 11. 验收失败必须如实报告

如果 Codex 在测试、浏览器验收、自动化操作、console 检查或功能点击中发现任何异常，必须如实报告，不允许把失败说成通过。

以下情况必须明确报告：

1. 浏览器 console 有 error / warning
2. 页面打不开或加载异常
3. 按钮点击没有反应
4. 功能结果没有更新
5. 自动化工具无法完成操作
6. 某项测试没有真正执行成功
7. 只能在某个浏览器正常、另一个环境异常
8. 看到异常但原因不确定
9. 回归样本没有跑完
10. 只是代码层检查通过，但真实浏览器未验证
11. 真实点击失败，但脚本或代码层验证通过

允许的表达：

- “这里有异常，我还不能确认原因”
- “这个功能在 Chrome 正常，但 Codex 内置浏览器异常”
- “我没有成功完成这项测试”
- “console 有 error，不能算通过”
- “这项验收结果不可靠，需要人工确认”
- “自动化点击没有成功触发真实用户操作”

禁止的表达：

- 明明看到按钮没反应，却写“功能正常”
- 明明 console 有 error，却写“console error 为 0”
- 自动化工具没点成功，却写“点击通过”
- 没跑完整回归，却写“回归通过”
- 用含糊表述掩盖失败
- 把环境异常、缓存异常、点击异常直接忽略

项目原则：

> 宁可报告一个后来证明无害的问题，也不要把真实 bug 藏进“验收通过”里。

Codex 的职责不是让结果看起来漂亮，而是帮助项目保持可控、可回滚、可验证。

## 12. 文件修改规则

不要在没有必要时大改多个文件。

不要顺手优化。

不要删除不确定用途的文件。

不要把 UI、数据、味觉逻辑、存档逻辑重新混在一个巨大文件里。

## 13. 版本号规则

版本号使用四段：`vA.B.C.D`。

- A = 测试 / 正式阶段
- B = 大功能阶段
- C = 重大逻辑 / 结构变化
- D = 小修补
