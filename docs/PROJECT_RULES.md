# 奶茶实验室 Project Rules

## 1. 项目定位

《奶茶实验室》是原创饮品研发经营模拟游戏。灵感来自《疯狂摇摇杯》，但不能复刻其原名、素材、UI、配方表或原文案。

最终目标是手游 + Steam 端游。当前 Web 技术只是快速原型手段。

《奶茶实验室》不是手忙脚乱接单摇奶茶的游戏，而是饮品研发、配方实验、试喝吐槽、经营验证的游戏。

## 2. 当前阶段

当前处于 v0.0.4.x 工程地基阶段，只做模块化、文档、回滚、测试规则和工程整理。

当前阶段仍是配方实验室测试版，暂时不做营业系统、员工系统、隐藏配方系统。

## 3. v0.0.4.x 禁止事项

- 不改玩法
- 不改味觉
- 不改评分
- 不改文案
- 不改 UI 视觉
- 不改交互
- 不新增原料
- 不新增隐藏配方
- 不新增营业系统

## 4. 配方系统长期规则

配方系统是成分池 + 比例系统，不是每类只能选一个。

茶底和乳基分开，“奶茶”不是茶底，而是茶 + 乳基的结果。

味觉审判优先级固定为：

```text
极端比例事故 > 质地事故（奶脂过载/吸管阻力） > 冲突组合 > 正常好组合 > 普通分类
```

隐藏配方以后走身份识别线，不是免死金牌；极端事故仍然应该优先被识别。

测试期可以显示完整属性条；正式版默认不展示所有后台数值。

宽屏桌面优先，竖屏只是兼容。

## 5. Vibe Coding 工作流

用户不是程序员。Codex 的任务单必须清晰、可执行、低歧义。每次修改后必须输出变更摘要和自检结果。

任务单应尽量包含：

- 版本号
- 阶段名
- 目标
- 本轮不做事项
- 范围边界
- 执行步骤
- 自检清单
- 回滚 / 备份要求

## 6. 反 if 地狱规则

味觉系统未来必须优先数据驱动，避免大量具体配方硬编码 if/else。

新增味觉规则时，优先放入：

- 原料属性库
- 协同 / 冲突规则
- 饮品模式规则
- 金标样本
- 反馈文案库
- 客群偏好数据

如果必须新增硬编码特判，必须在报告里说明原因和数量。

## 7. 稳定版本规则

每个稳定版本需要：

- 更新 VERSION_LOG
- 更新 AI_CONTEXT
- 确认 git status
- 提交 commit
- 打 tag
- 必要时做本地备份
- 输出可读的测试报告

## 8. 自动化验收与失败自救规则

Codex 每次完成版本修改后，应优先尝试自动化验收，尽量减少用户手动测试负担。

如果第一次自动化验收失败，Codex 不应立刻把全部测试交给用户，而应先自行做一次低风险恢复尝试，例如：

- 确认本地静态服务是否仍在运行
- 重启静态服务
- 换一个可用端口
- 确认访问 URL 是否正确
- 重新打开页面
- 重连可用的浏览器自动化工具

如果低风险恢复后仍失败，Codex 才应报告：

- 失败原因
- 已尝试的恢复步骤
- 已完成的替代检查
- 用户仍需手动检查的最小 P0 清单

## 9. 文件修改规则

不要在没有必要时大改多个文件。

不要顺手优化。

不要删除不确定用途的文件。

不要把 UI、数据、味觉逻辑、存档逻辑重新混在一个巨大文件里。

## 10. 版本号规则

版本号使用四段：`vA.B.C.D`。

- A = 测试 / 正式阶段
- B = 大功能阶段
- C = 重大逻辑 / 结构变化
- D = 小修补
